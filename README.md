# Фабрика гипотез

Интеллектуальный инструмент, который по отчёту о хвостах обогатительной фабрики
формирует ранжированный список **проверяемых инженерных гипотез** по снижению
потерь ценных металлов, с обоснованием, ссылками на литературу, оценкой новизны,
рисками, ожидаемой ценностью и дорожной картой проверки.

## Ключевая идея

Система не просто «спрашивает модель». Сначала — **детерминированный
количественный диагноз** баланса хвостов, затем генерация:

1. **Диагноз без ИИ.** Парсер разбирает Excel-баланс (классы крупности,
   минералогия, извлекаемый/неизвлекаемый металл) и по прозрачным правилам
   определяет драйверы потерь:
   - закрытые сростки и крупные раскрытые зёрна → проблема раскрытия →
     **измельчение/классификация**;
   - раскрытый металл в тонких классах (шламы) → проблема **флотации**.
   Каждый сигнал измеряется в тоннах извлекаемого металла.
2. **RAG.** Гибридный поиск (BM25 + опциональные эмбеддинги) по корпусу учебников
   и корпусу известных решений подбирает подтверждающие выдержки с цитатами.
3. **Генерация.** По каждому сигналу формируется структурированная гипотеза со
   всеми полями: обоснование, источники, механизм влияния, новизна, риски,
   ожидаемая ценность (KPI), дорожная карта проверки.

Всё интерпретируемо и валидируемо экспертом.

Дополнительно: при загрузке можно задать **целевую задачу и ограничения**
(например, «без замены крупного оборудования») — генерация учтёт их; после
анализа доступен **диалог** для уточнения и доработки гипотез (ответы опираются
на диагноз, профиль оборудования фабрики и литературу с цитатами); базу знаний
можно **пополнять своей литературой** (PDF/DOCX/TXT) — документы индексируются
на лету и цитируются наравне со вшитым корпусом.

## Быстрый старт (Docker)

```bash
docker compose up --build
# открыть http://localhost:8000
```

Сервис работает **без внешних ключей** в базовом режиме (диагноз + поиск по
литературе + гипотезы). Для расширенных формулировок задайте доступ к LLM:

```bash
YANDEX_API_KEY=... YANDEX_FOLDER_ID=... docker compose up --build
```

### Запуск готового образа (без сборки)

```bash
docker compose -f docker-compose.deploy.yml up -d
# образ: ghcr.io/gleb-vlasov/hypofactory:latest (публичный, ~500 МБ, CPU-only)
```

## Локальный запуск (без Docker)

```bash
cd backend
pip install -r requirements.txt
export PYTHONPATH="$PWD"
uvicorn hypofactory.api.app:app --host 0.0.0.0 --port 8000
```

## Конфигурация (переменные окружения)

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `YANDEX_API_KEY` / `YANDEX_FOLDER_ID` | Доступ к LLM (OpenAI-совместимый API) | — (базовый режим) |
| `LLM_MODEL_ID` | Модель генерации | `qwen3-235b-a22b-fp8` |
| `VISION_MODEL_ID` | Vision-модель для расшифровки схем/регламентов (тот же эндпоинт и ключ) | `qwen3.6-35b-a3b` |
| `LLM_TEMPERATURE` | Температура | `0.2` |
| `EMBEDDINGS_ENABLED` | Семантический слой RAG (гибридный поиск BM25+вектора; без ключа/индекса деградирует до BM25) | `true` |
| `APP_ACCESS_KEY` | Разграничение доступа: все `/api/*` требуют заголовок `X-API-Key` (веб-интерфейс запросит ключ сам) | — (выключено) |
| `METAL_PRICE_28_USD_T` / `METAL_PRICE_29_USD_T` | Референсные цены металлов для экономической оценки гипотез, USD/т (`0` — отключить) | `16500` / `9200` |
| `DATA_INDEX_DIR` | Каталог индекса знаний | `data_index` |

Шаблон — `backend/env.example` (скопировать в `backend/.env`).

## API

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/health` | Статус: режим, размер корпуса, готовность LLM |
| `POST` | `/api/analyze` | Excel-баланс (+ `goal`: целевая задача; + `images`: до 5 изображений схем/регламентов — расшифровываются vision-моделью и учитываются в гипотезах) → диагноз + гипотезы (JSON) |
| `POST` | `/api/export/{docx\|pdf\|csv\|tasks\|protocol}` | Экспорт: отчёты (DOCX/PDF/CSV), задачи для импорта в Jira/YouTrack (CSV), протоколы лабораторных экспериментов (DOCX) |
| `POST` | `/api/chat` | Диалог по результатам: уточнение и доработка гипотез с опорой на диагноз, профиль фабрики и литературу |
| `GET` | `/api/knowledge` | Состав базы знаний (вшитый корпус + загруженные документы) |
| `POST` | `/api/knowledge` | Добавление литературы (PDF/DOCX/TXT/MD) или изображения схемы/регламента (PNG/JPG/WebP/BMP) в базу знаний на лету |
| `POST` | `/api/feedback` | Оценка гипотезы экспертом |

Веб-интерфейс доступен на корне (`/`).

## Архитектура

```
backend/hypofactory/
  ingestion/   парсинг Excel-баланса, PDF-литературы и изображений схем (vision)
  diagnosis/   детерминированный движок сигналов потерь
  rag/         корпус, BM25, эмбеддинги, гибридный ретривер
  generation/  схема гипотезы, промпты, генератор (LLM + шаблонный фолбэк)
  llm/         провайдер-агностичный клиент (OpenAI-совместимый)
  api/         FastAPI (эндпоинты + отдача статики)
frontend/      веб-интерфейс без сборки (HTML/CSS/JS)
data_index/    предпосчитанный корпус знаний (corpus.jsonl)
scripts/       сборка индекса и проверочные прогоны
```

## Соответствие функциональным требованиям

| Требование | Реализация |
|---|---|
| Парсинг текстовой литературы | вшитый корпус (учебники, экспертные решения) + загрузка PDF/DOCX/TXT/MD через UI/API на лету |
| Импорт структурированных данных | Excel-баланс испытаний (потоки, классы крупности, минералогия, извлекаемость) |
| Обработка изображений | схемы цепочек аппаратов, регламенты, списки оборудования (PNG/JPG/WebP/BMP): vision-модель расшифровывает в структурированный текст (тип, оборудование, цепочка, режимы, выводы) — и в базу знаний, и как контекст конкретного анализа |
| Метаданные источников | авторы/год/условия экспериментов при загрузке литературы; попадают в цитирование; дата добавления |
| Извлечение сущностей и связей | парсер строит модель «поток → класс → минеральная форма → извлекаемость»; связи видны в графе |
| Паттерны и пробелы в знаниях | детерминированные сигналы потерь (паттерны); оценка новизны против известных решений (пробелы) |
| Аналогии / контрфактуальность | RAG-аналогии из литературы; гипотезы — контрфактуальные вмешательства («что изменить → какой эффект»); переработка по ограничениям |
| Ранжирование по критериям | композитный приоритет: 55% потенциальный эффект (т) + 20% новизна + 25% реализуемость/уверенность (учитывает риски и вердикты эксперта) |
| Текстовое обоснование | у каждой гипотезы, с числами диагноза |
| Привязка к источникам | название, страница, цитата |
| Графы и диаграммы | диаграмма потерь по классам крупности; интерактивный граф связей «поток → класс → гипотеза» |
| Оценка неопределённости | confidence (0–1) у каждой гипотезы + дорожная карта верификации с критериями успеха/провала |
| Бизнес-отчёты PDF/DOCX | с ранжированием и KPI |
| Экспорт задач | CSV для Jira/YouTrack, JSON, открытый REST API |
| Обратная связь и обучение | вердикты ✓/✗ влияют на следующие анализы; переработка гипотез моделью по указанию эксперта |

## Нефункциональные характеристики

- **Интерпретируемость.** Диагноз детерминирован (прозрачные правила, числа из
  баланса), каждая гипотеза обоснована этими числами и цитатами; граф связей
  показывает путь «сигнал → гипотеза»; формула приоритета раскрыта в интерфейсе.
- **Мультиязычность.** Литература на русском, английском и китайском: лексический
  индекс токенизирует латиницу и CJK (биграммы), семантический слой и
  LLM-генерация мультиязычны; вывод — всегда на русском.
- **Надёжность.** Устойчивый парсер (шумные/неполные файлы → анализ по доступной
  части с предупреждениями), деградация LLM/эмбеддингов без падения, явная
  обработка противоречий в источниках.
- **Производительность.** Базовый режим — секунды; полный LLM-режим — порядка
  минуты (генерация по сигналам распараллелена); с приложенными схемами —
  ~2–3 минуты (расшифровка изображений тоже распараллелена).
- **Безопасность.** Полностью локальное развёртывание (наружу уходит только
  анонимизированный запрос к LLM API; базовый режим — вообще без внешних
  вызовов); разграничение доступа через `APP_ACCESS_KEY`; секреты — через
  переменные окружения; TLS — терминацией на реверс-прокси.

## Расширение на новые предметные области

Ядро (конвейер `диагноз → поиск → генерация → критика → отчёт`) доменно-нейтрально;
предметная область задаётся тремя заменяемыми компонентами:

1. **Корпус знаний** — `data_index/corpus.jsonl` (+ векторный индекс): литература
   домена. Пересобирается скриптом из любого набора PDF/DOCX; пользовательская
   литература догружается через UI/API без пересборки.
2. **Диагностические правила** — `diagnosis/engine.py`: чистые функции
   «структура данных → сигналы проблем». Для нового домена (металлургия, полимеры,
   композиты) пишется свой модуль правил с тем же выходным контрактом `ProblemSignal`.
3. **Доменный профиль генерации** — системный промпт, профиль оборудования
   (`data_index/facility_profile.md`) и шаблоны фолбэка.

Схема гипотезы (обоснование, источники, механизм, новизна, риски, ценность,
дорожная карта), RAG, LLM-клиент, API, UI, экспорт и цикл обратной связи
переиспользуются без изменений.

## Устойчивость к неполным данным

Парсер баланса привязывается к текстовым маркерам, а не к фиксированным адресам
ячеек: переживает `#REF!`, варианты меток классов, пропущенные значения и строки.
При неполном файле анализ выполняется по доступной части, предупреждения
показываются в интерфейсе и возвращаются в JSON (`report.warnings`). При
недоступности LLM или эмбеддингов система деградирует до диагноза + BM25 +
шаблонных гипотез, но не падает.

## Требования к окружению

- CPU-only, без GPU. ~1–2 ГБ RAM для базового режима.
- Интернет нужен только для обращения к LLM API (генерация); диагноз и поиск по
  литературе работают полностью локально.
- Пересборка индекса знаний: `python scripts/build_index.py`.

## Пересборка индекса знаний

```bash
DATA_DIR="/путь/к/материалам" python scripts/build_index.py
```

Индекс (`data_index/corpus.jsonl`) уже входит в репозиторий, поэтому образ
собирается без доступа к исходным материалам.
